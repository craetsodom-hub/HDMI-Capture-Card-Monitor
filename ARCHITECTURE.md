# Architecture

## Foundation

The application is stable .NET 10 WPF, x64-only, using MVVM through `CommunityToolkit.Mvvm`. Media Foundation is the primary discovery and Phase 2A preview backend. Required Media Foundation, Direct3D 11, DXGI, and Win32 APIs are generated by CsWin32; no third-party native wrapper or codec runtime is used. DirectShow remains only a future, narrowly scoped compatibility candidate.

## Boundaries

- Device enumeration: enumerate video/audio devices and capabilities.
- Capture backend: negotiate and own an active device stream.
- Video preview renderer: present negotiated video samples.
- Audio monitor: route audio samples to local playback.
- Recording service: own optional recording pipeline and output lifecycle.
- Device recovery service: coordinate device-loss detection and recovery.
- Capture diagnostics: produce safe technical observations and support data.

Phase 1 introduced `ICaptureDeviceDiscoveryService`. Phase 2A introduces `ICapturePreviewService` and `IPreviewSurface`, immutable preview request/result/failure/diagnostic models, a Media Foundation session owner, a D3D renderer, a dedicated `HwndHost`, and a bounded diagnostics tracker. Device discovery, preview coordination, native frame acquisition, GPU rendering, the WPF view model, and diagnostics remain separate responsibilities. View models orchestrate state and user intent but never own COM objects, textures, or symbolic-link behavior.

## Threading, lifetime, and performance

Every asynchronous operation must take a cancellation token where appropriate. UI updates marshal to the WPF dispatcher; device and media work stays off the UI thread. Native COM, streams, device handles, and cancellation registrations have one clear owner and are deterministically disposed in the reverse order of acquisition. Shutdown is idempotent.

Phase 2A owns one explicitly MTA native preview thread per active session. That thread performs synchronous `IMFSourceReader.ReadSample`, renders the returned GPU sample immediately, releases it before the next request, and owns all Media Foundation and D3D cleanup. There is no application frame queue. Stop cancellation uses a separately COM-initialized MTA control path to call `Flush`; the preview thread remains the sole native owner and shutdown waits at most three seconds without releasing objects underneath a blocked worker. Cleanup is layered so every native release and COM uninitialization is attempted even if an earlier step or logger fails, and completion is signalled in the outermost `finally`. OpenCV and managed frame copies are not part of the preview path.

## Errors, logs, settings, and tests

Backends return typed, contextual failures upward; the UI translates them into plain-language status while diagnostics retains safe technical detail. There are no silent fallbacks: choosing DirectShow must be observable in logs and diagnostics. Local logging supports Debug, Information, Warning, and Error. It stores text only and excludes media and sensitive user data. Logger creation is centralized: if local files cannot be created, a local no-op logger is used and the shell displays a non-fatal notice. File names are collision-resistant and retention is bounded to matching application logs.

The state model permits `Previewing → Recording → Previewing`; stopping recording must not imply stopping the live preview.

## Phase 1 discovery

Phase 1 uses generated CsWin32 Media Foundation bindings behind `ICaptureDeviceDiscoveryService`. Native work runs synchronously on background COM-initialized workers and returns immutable managed device and capability models. Symbolic links are opaque identities and never appear in UI or logs. The service does not read frames or render video.

Settings will be local, versioned, validated, atomic on write, and Store-compatible. Hardware and native integration stay behind interfaces; state, capability decisions, validation, settings serialization, and error mapping are unit-testable without hardware. Hardware tests remain manual/integration tests in the QA matrix.

## Phase 2A GPU preview

The selected Phase 1 capability is re-resolved by native media-type index and verified for size, frame-rate rational, subtype, and interlace information. `IMFSourceReaderEx.SetNativeMediaType` applies that exact native mode. Reader attributes request low latency, hardware transforms, source disconnection on shutdown, and the session's `IMFDXGIDeviceManager`. Output negotiation requests GPU-backed NV12; compressed MJPEG is decoded by Media Foundation rather than application code.

`D3D11PreviewRenderer` creates a hardware D3D11 device with BGRA and video support, falling back to WARP only as an observable degraded mode. Progressive acquisition uses scoped reverse-order ownership, so a partial hardware attempt is fully released before WARP and a partial WARP attempt is also released. An `ID3D11VideoProcessor` consumes each `IMFDXGIBuffer` texture and writes directly to a BGRA8 two-buffer flip-discard swap chain. Maximum frame latency is one. Each frame clears black and uses a centered Fit rectangle, so unused pixels become letterbox or pillarbox bars. Resize requests use a capacity-one mailbox and are executed only by the preview thread.

The `HwndPreviewSurface` retains one child HWND created hidden, without `WS_VISIBLE`, and initially off-screen. Its WPF host is activated before D3D ownership begins and remains stable until that session's native cleanup completes; the child stays hidden until the first successful presentation. Window `Closing` synchronously stops/disposes preview before WPF destroys this host. This sequencing avoids Win32 airspace occlusion while inactive, stale-frame flashes, and presentation to a destroyed HWND.

The monotonic watchdog distinguishes sample arrival from presentation and observes whether the surface is presentable. Minimized or zero-sized surfaces may skip rendering while continuing to prove healthy input; visible presentation absence is a presentation failure, while sample absence is an input stall. Diagnostics report actual negotiated output fields, received/rendered FPS, sample timestamp cadence, bounded processing and sample-return-to-present average/p95, intentional surface skips, `DXGI_ERROR_WAS_STILL_DRAWING`, and safe failure categories. They do not claim camera-to-screen or HDMI latency.
